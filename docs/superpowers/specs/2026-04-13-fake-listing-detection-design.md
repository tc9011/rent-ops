# 假房源检测系统设计

**日期：** 2026-04-13
**状态：** Draft
**范围：** rent-ops Skill 增强 — 系统化假房源识别

---

## 背景

rent-ops 在小红书传播后，大量用户反馈担心碰到假房源。现有的风险检测能力散落在 `risk.md` 和 `auto-evaluate.md` 中，缺少系统化的假房源识别机制。

## 设计决策

- **方案选型：** 新建 `modes/verify.md` 集中管理所有检测逻辑（方案 A），不分散到多个 mode 文件
- **触发模式：** 评估时自动跑快检 + `/rent verify` 手动跑完整版
- **处理策略：** 分级处理 — 高风险硬拦截、中风险软警告

---

## 1. 检测分级

### 风险等级

| 等级 | 含义 | 判定条件 |
|---|---|---|
| `RED` | 高风险，疑似假房源 | 任一项命中红色规则 |
| `YELLOW` | 中风险，需留意 | 命中黄色规则 |
| `GREEN` | 无异常 | 未命中任何规则 |

### 综合判定

- 1+ 个 `RED` → 整体「高风险」→ 硬拦截，提示用户确认是否继续评估
- 2+ 个 `YELLOW` → 整体「中风险」→ 软警告，报告中醒目标注
- 其余 → 「低风险」→ 正常评估

### 两种触发模式

| 模式 | 触发方式 | 检测范围 | 耗时 |
|---|---|---|---|
| 快检（lite） | `/rent {url}` 评估时自动触发 | 通用层 5 项（纯规则 + 评估已有的 WebSearch） | 快 |
| 完整（full） | `/rent verify {url}` 手动触发 | 通用层 + 平台专属层，全部 11 项 | 较慢（多次 WebSearch） |

## 2. 检测规则

### 2.1 通用检测层（所有平台，快检 + 完整版都跑）

| # | 检测项 | RED（高风险） | YELLOW（中风险） |
|---|---|---|---|
| 1 | 价格异常 | 低于商圈均价 30%+ | 低于均价 15-30% |
| 2 | 隔断检测 | 一居室 < 15㎡ / 两室 ≤ 30㎡ | 一居室 15-20㎡ |
| 3 | 信息一致性 | 标题户型/价格与正文矛盾（如标题写整租正文写合租） | 描述模糊、关键信息缺失 |
| 4 | 联系方式风险 | 要求转账/押金才看房、无任何联系方式 | 只留微信不留电话 |
| 5 | 挂牌时长 | 挂牌超过 90 天未下架 | 挂牌 45-90 天 |

### 2.2 平台专属检测层（完整版才跑，按 URL 域名自动选择）

**豆瓣（douban.com）：**

| 检测项 | RED | YELLOW |
|---|---|---|
| 发帖人主页分析 | 注册 < 7 天 + 仅房源帖 / 多城市发房源 | 注册 < 30 天 |
| 评论区揭露 | 评论有"中介""假房源""骗子" 3+ 条 | 1-2 条质疑 |
| 文案模板 | 全文套话无个性化细节 | 部分模板化 |

**小红书（xiaohongshu.com）：**

| 检测项 | RED | YELLOW |
|---|---|---|
| 评论区翻车 | "去了不是这套""和图片不符" 3+ 条 | 1-2 条 |
| 博主主页 | 近 30 条笔记全是房源 | 房源占比 > 70% |
| 图片风险 | 带其他平台水印 / 明显精修 | 仅有官图无实拍 |

**贝壳（ke.com）：**

| 检测项 | RED | YELLOW |
|---|---|---|
| 经纪人挂房量 | 同一经纪人 50+ 套在挂 | 30-50 套 |
| 房源状态 | 页面 404 / 已下架但仍可搜到 | 上架超 60 天未更新 |

**58同城/安居客（58.com / anjuke.com）：**

| 检测项 | RED | YELLOW |
|---|---|---|
| 同图检测 | 同图出现在 3+ 条不同房源 | 2 条 |
| 刷新频率 | 每天刷新 | 2-3 天刷新一次 |
| 认证状态 | 未认证 + 低价 | 未认证 |

**自如（ziroom.com）：**

| 检测项 | RED | YELLOW |
|---|---|---|
| 用户差评 | WebSearch 到 3+ 条该房源差评 | 1-2 条 |

### 2.3 检测项说明

- **价格变动异常（原第 9 项）** 合并到「价格异常」中 — 如果 tracker 中有该房源的历史价格记录，对比变动幅度（7 天降 30%+ → RED，14 天降 15-30% → YELLOW）。首次评估无历史数据则跳过此子项。
- **二房东检测（原第 2 项）** 合并到平台专属层的「发帖人主页分析」和「经纪人挂房量」中，按平台用不同方式实现。
- **首次发帖账号（原第 12 项）** 合并到豆瓣「发帖人主页分析」中。
- **文案模板检测（原第 10 项）** 靠 agent 语言理解能力判断，不需要额外工具调用。
- **图片反查（原第 6 项）** 合并到小红书「图片风险」和 58「同图检测」中，按平台特点实现。

## 3. 输出

### 3.1 卡片视图集成（listings-view.html）

每张房源卡片增加风险展示：

- **高风险：** 卡片边框变红，右上角标 `🔴 疑似假房源`
- **中风险：** 卡片边框变黄，右上角标 `🟡 需留意`
- **低风险/通过：** 不额外标注

卡片底部新增风险摘要行，如：
```
🔴 价格偏离均价38% · 发帖人注册5天
```

### 3.2 数据层（listings.json）

每条房源新增 `verify` 字段：

```json
{
  "name": "xxx小区",
  "price": 2800,
  "verify": {
    "level": "red",
    "signals": [
      {"rule": "price_anomaly", "level": "red", "detail": "低于均价38%"},
      {"rule": "new_account", "level": "red", "detail": "注册5天，仅房源帖"}
    ]
  }
}
```

## 4. 评估流程对接

### auto-evaluate.md 修改

在步骤 0（抓取房源信息）和步骤 1（红线检查）之间插入步骤 0.5：

```
步骤 0   — 抓取房源信息
步骤 0.5 — 快检（verify lite 模式）
  → 高风险：硬拦截，提示用户确认是否继续
  → 中/低风险：继续评估
步骤 1   — 红线检查
步骤 2   — 八维度评分
```

### 评分维度复用

`_shared.md` 中两个评分维度直接引用 verify 结果，不重复搜索：

- **风险维度：** verify 通用检测结果 → 映射为 1-5 分
- **房东/中介可靠性维度：** verify 的发帖人分析 + 联系方式风险 → 映射为 1-5 分
